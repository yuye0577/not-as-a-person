## Event ID 命名规则

- 格式：`<分类前缀>-<三位序号>`，例如 `EAT-001`、`EXP-003`。
- 序号在分类内递增、不复用——即使以后删除某个事件，也不回收它的编号，避免历史引用（比如某次 Today's Memory 的日志、或未来的数据统计）对不上号。
- 代码里直接消费三个核心字段：
  - `event.id`　　　　→ 如 `"EXP-003"`
  - `event.category`　→ 如 `"Explore"`
  - `event.weight`　　→ 由"稀有度"+"适用动物星级"共同换算出的抽样权重（见下文）

## 事件字段规格（Schema）

每个事件条目固定包含以下字段，写作时按此顺序填写：

| 字段 | 说明 |
|---|---|
| Event ID | 唯一编号，见上方命名规则 |
| 名称 | 人类可读的标题，仅用于文档/后台维护，不会出现在玩家界面上 |
| 世界层 | 客观发生的事实，玩家看不到，仅供写作时对照（呼应 DESIGN.md 第7.1节） |
| 感知层 | 玩家实际看到的感官文案，按猫/狗/兔子分别书写（呼应 DESIGN.md 第8节 Animal Perspective，同一个世界层事实，不同动物的感知不同） |
| 选择 | 2～3 个本能反应选项（A/B/C） |
| 反馈 | 每个选择对应的"当下感受"文案，现在时 |
| 状态影响 | 每个选择对三层状态（生理/情绪/社会，见 DESIGN.md 第6节）的具体数值变化 |
| 稀有度 | Common / Uncommon / Rare / Secret 四档之一（见下方"稀有度分层"） |
| 适用动物 | 猫/狗/兔子各自的相关度星级（1～5★），用作该物种抽到这个事件的权重 |
| 触发时间 | Morning / Noon / Evening / Night，可多选，表示这个事件倾向在哪些时段被调度 |
| 天气 | Sunny / Rain / Wind / Snow / Any，事件生效所需的天气条件 |
| 前置条件 | 触发这个事件所需满足的状态阈值，例如 `Hunger > 70`、`Bond > 50`；无要求写"无" |
| 情绪标签 | 这个事件代表的**主导情绪**（单个词，如 Curiosity / Fear / Relief），供 Today's Memory 统计使用（见下文） |

> ⚠️ **新增前置需求**：「天气」字段意味着事件系统需要一个"当天天气"状态，这是 DESIGN.md 目前还没有的概念。实现前需要先补一个简单的"每日天气随机量"（例如开局时随机决定今天是 Sunny/Rain/Wind/Snow 中的一种），事件调度时按这个天气过滤候选事件。

## 稀有度分层（Rarity Tiers）

**"分类"（Eating/Play/…/Rare）与"稀有度"（Common/Uncommon/Rare/Secret）是两个独立的轴**：分类回答"这是关于什么的事件"，稀有度回答"这个事件多久出现一次"。

⚠️ 容易混淆的地方：**"Rare 稀有事件"这个分类**，不等于**"稀有度 = Rare"这个分层**——Rare 分类里的事件通常会标"Rare"或"Secret"稀有度，但两者是两套独立标签，代码里对应 `event.category` 与 `event.rarity` 两个不同字段；反过来，Eating/Play 等其他分类里的事件，也完全可以标注 Uncommon/Rare 稀有度（比如"主人今天请假"更适合归在 Owner 分类，但稀有度是 Rare）。

| 稀有度 | 抽样权重 | 说明 | 示例 |
|---|---|---|---|
| Common | 60% | 几乎每天都会遇到的日常小事 | 闻到饭香、晒太阳、打哈欠 |
| Uncommon | 25% | 偶尔出现，让每天略有不同 | 快递来了、有陌生人、主人打电话 |
| Rare | 10% | 比较少见，算是"今天运气不错/不好" | 主人今天请假、家里来了客人、突然下雨 |
| Secret | 5% | 刻意隐藏、很少出现，专门留给二周目/多次游玩的玩家做惊喜 | 梦境、半夜月光、主人哭了、窗外烟花 |

抽样时按这个权重分布，从 DESIGN.md 第7.4节设定的"单局 30～40 个事件"里决定每档大致抽多少个，而不是四档等概率随机。

## 情绪标签与 Today's Memory 的联动

每个事件的「情绪标签」字段，会在一天结束时被统计，用来让 Today's Memory（DESIGN.md 第10节）不只是数值判断，而是真正像是在回忆这一天：

- 记录当天每个已发生事件的情绪标签，按出现频次排序。
- 生成两类回忆开场句：
  - **高频情绪**：如果某个情绪标签出现次数明显最多，生成"今天，你一直在{情绪}这个世界。"这样的整体基调句（例如出现最多的是 Curiosity → "今天，你一直在好奇这个世界。"）。
  - **高光时刻**：从当天事件里挑一个"独一无二"的情绪时刻（例如全天唯一一次 Relief 或 Fear），生成"今天，你最{情绪对应的形容词}的时刻，是……"这样的单点句（例如唯一一次 Relief 对应"熟悉的脚步声" → "今天，你最安心的时刻，是听见主人回家的脚步声。"）。
- 这两句可以和 DESIGN.md 第10.3节已有的"挑选真实发生过的片段"逻辑拼在一起使用：原有逻辑负责挑"发生了什么"，情绪标签负责挑"感觉像什么"。

## Owner Event Design Guideline（Owner 类事件设计经验）

这不是 DESIGN.md 第12章那种全局文案规范，而是基于 OWN-001～006 六个事件加上 Batch 1A QA 的实际发现，专门针对 **Owner 类别**总结出的设计经验。以后新增/修改任何 Owner 事件，先对照这份经验，再对照 DESIGN.md 第12章和下方的 Checklist。

### 1. 应该避免的重复

对 OWN-001～006 的"世界层事实"做了一次分类：

| 事件 | 世界层事实 | 归类 |
|---|---|---|
| OWN-001 | 主人回家 | 到家/离开 |
| OWN-002 | 主人打电话，情绪不同 | 情绪信号 |
| OWN-003 | 主人准备出门 | 到家/离开 |
| OWN-004 | 主人离开后的安静 | 到家/离开 |
| OWN-005 | 被摸/顺毛 | 亲密接触 |
| OWN-006 | 主人静止不动，可挨近 | 亲密接触 |

**真正重复的风险**：不是"同一个话题出现两次"，而是"同一个世界事实被切成好几个几乎等价的切片"。OWN-003（离开前）和 OWN-004（离开后）是同一个"主人离开"事实的两个时间切片，这次是可以接受的——因为切片本身有明确的先后关系，读起来是自然的前后两拍，不是重复。但**这个技巧不能反复用在同一个事实上**：不应该再为"主人离开"这件事写第三个切片（比如"主人离开很久了"），否则就会变成真正的重复。

**同主题不同角度（可以保留的模式）**：OWN-001（到家瞬间的强互动）vs OWN-006（日常状态下的静态陪伴）——都是"亲密接触"主题，但强度、诱因、数值权重都不同，这是合理的角度差异，不是重复。

**以后应该避免**：
- 不要再写第三个"到家/离开"类事件，除非它是明显不同的角度（比如"主人这次带着明显的伤心情绪回来"这种带情绪反差的到家，而不是又一次单纯的到家/离开）。
- 不要再写一个和 OWN-005 几乎等价的"被摸"事件（比如"挠痒痒"），除非触感/情境有本质区别。
- Batch 2 计划中的 OWN-007（"比平时更晚回来"）要特别小心——这本质上是 OWN-001（到家）的第三个变体，写的时候必须让"迟到"本身带来的情绪（等待、不确定）成为主角，而不是又写一遍"到家真开心"。

### 2. Choice Pattern 普查

> 这一节的"Choice Pattern"就是 DESIGN.md 第14.1节正式定义的 **Experience Pattern** 概念在 Owner 类别里的一次预演——当时只在 Owner 内部统计，后来 Content QA v2 把它扩展成跨全部事件的统一概念和常规审查项（第14.5节 Experience Pattern Audit）。以后写 Owner 事件，除了看这一节的 Owner 内部统计，也要看 Experience Pattern Audit 的全局基线，两者结论一致时优先级更高。

逐个拆解 OWN-001～006 的选项结构：

| 事件 | Choice Pattern |
|---|---|
| OWN-001 | 靠近亲密接触 / 保持距离观察 / 主动回避躲藏（3选） |
| OWN-002 | 靠近陪伴 / 保持自己节奏 |
| OWN-003 | 靠近跟随 / 不靠近 |
| OWN-004 | **等待守候 / 转移注意力自己找事做** |
| OWN-005 | **主动回应互动 / 被动接受** |
| OWN-006 | 靠近陪伴 / 保持原地独立 |

**"靠近 vs 保持距离/独立"这个形状，6 个事件里用了 4 个**（001/002/003/006），已经明显过多。真正跳出这个形状的只有两个：OWN-004 的"等待 vs 自己找事做"（这不是空间距离选择，是"抱着期待 vs 主动转移注意力"的心理选择），和 OWN-005 的"主动回应 vs 被动接受"（这里根本没有"距离"这件事，互动已经发生，选择的是参与程度）。

**几乎没出现的 Pattern**（用户提到的几种里，Owner 目前一个都没有）：
- 主动探索 vs 继续观察（EXP 类常用，Owner 还没用过）
- 立即行动 vs 再等等（一个纯粹的"时机"选择，不涉及靠近谁）
- 留在原地 vs 去另一个地方（不是"要不要靠近主人"，是"要不要换个空间"）

**Batch 1B 及以后 Owner 事件的优先级**：优先复用"等待/守候"型和"主动回应/被动接受"型，其次可以尝试"立即行动 vs 再等等"，**除非有明确理由，暂停再写"靠近 vs 保持距离"这个形状**——不是禁止，是这个形状的配额已经用满，需要先靠别的形状把 Owner 类的选择质感撑开。

### 3. 情绪节奏

OWN-001～006 的 emotionTag：Relief（001）/ Concern（002）/ Concern（003）/ Serenity（004）/ Comfort（005）/ Comfort（006）。

Concern 和 Comfort 各出现了两次，且后两个（005/006）连续都是 Comfort，整体基调偏温暖安稳。**完全没有覆盖**：好奇（Curiosity）、犹豫/警觉（Wariness/Alertness）、失落（Disappointment）、期待（Expectation，虽然在 Eating 类用过）。

**关于"想念"**：这是用户提到的一种很适合 Owner 类的情绪，但目前 `EVENT_EMOTION_ZH` 的 15 个标签里没有严格对应的词——最接近的是 Concern（已经用了两次）。**这里不建议现在新增情绪标签**（这是数据层面的小扩展，但本项目当前阶段明确"停止新增 Framework，重心转向 Content"，新增标签值虽然成本低，仍然算是往系统层面加东西，先记录成 Future Idea，不在这轮处理）。以后如果 Owner 类"思念/等待"题材写多了，成为一个明显的空缺，再回来统一评估是否值得加一个标签。

**建议**：接下来的 Owner 事件优先安排 Curiosity、Wariness、Expectation、Alertness 里的一到两个，避免第三次出现 Comfort 或 Concern 连续堆叠。

### 4. 感知通道普查

| 事件 | 主要通道 | 次要通道 |
|---|---|---|
| OWN-001 | 听觉（钥匙声/脚步声） | 身体反应（狗尾巴） |
| OWN-002 | 听觉（语调变化） | — |
| OWN-003 | 听觉（响动） | 身体反应 |
| OWN-004 | 听觉（声音的缺席） | 身体反应/内感受（心跳、呼吸） |
| OWN-005 | **触觉**（唯一一个） | — |
| OWN-006 | 视觉（观察静止的人） | 身体反应 |

**听觉在 6 个事件里出现了 4 次**，且集中在"脚步声/钥匙声/响动"这几个具体意象上——这正是用户点名担心的问题。触觉只出现过 1 次（OWN-005），嗅觉、光线、温度、气流、地板震动、衣物摩擦声、"熟悉的气味"作为独立通道**一次都没出现过**。

**建议**：Owner 类接下来应该优先补嗅觉相关内容——比如通过"主人身上的气味变化"来感知信息，这本来就该是 Owner 类最有潜力、却完全没用过的通道（尤其是狗，嗅觉本该是它最主要的感知方式，但 Owner 类目前全部靠狗的听觉/尾巴反应，嗅觉完全没派上用场）。其次可以尝试温度（主人身上带进来的室外温度）、地板震动（脚步的震动感，区别于"脚步声"这个听觉描述）。

### 5. 动物差异总结

**猫**：已经用得比较好的是"不动+简短反应"（观察后不主动、简短一瞥后转移注意力），呼应"谨慎、爱观察"的设定。风险：003 和 006 两次都是这个模式，如果继续这样写会变成猫的"标准动作"而不是真差异——以后可以尝试猫"表面冷淡，选择后才展现真实依赖"这种反差写法。

**狗**：已经用得好的是"身体先于思考反应"（尾巴摇动、兴奋的生理信号）和"焦躁踱步"（分离焦虑的外显行为，OWN-004 新加的）。**明显的缺口**：狗最该用上的嗅觉，在 Owner 类 6 个事件里一次都没出现——目前全靠"尾巴"这一个信号在承担狗的差异化，风险是这个信号本身也会被用滥。

**兔子**：已经用得好的是"生理性应激反应"（浑身一紧/缩紧）和"从藏身处探头，不完全暴露"。**明显的缺口**：兔子目前全部是紧张/警惕类反应，没有一次展现"建立信任后放松下来"的对照面——兔子应该也有机会在羁绊值高的情境下表现出不同于"一直紧绷"的状态，否则兔子的行为谱系会显得比猫狗更单一。

**核心提醒**：三种动物的差异不应该靠"换主语+换几个形容词"实现，而应该来自**不同的感知优先级**（狗先用鼻子，猫先用耳朵/眼睛，兔子先用耳朵且伴随生理应激）和**不同的行为逻辑**（猫倾向观察后再决定、狗倾向身体先动、兔子倾向先僵住再决定）。写完之后可以做一个简单测试：把三句感知层文案里的动物名字互相调换，如果调换后读起来毫无违和感，说明这次的差异化还不够。

### 6. Owner Event Checklist

以后每写一个新的 Owner 事件，落笔前后对照：

- [ ] 是否和已有 Owner 事件的"世界层事实"撞题？如果是同一事实的不同时间切片，是否只切了一次（没有切第三刀）？
- [ ] Choice Pattern 是否和最近几个 Owner 事件重复？"靠近 vs 保持距离"这个形状是否已经用得够多，需要换一种？
- [ ] 三个动物的感知层，是否体现了真正不同的感知优先级和行为逻辑，而不是同一句话换主语？（自测：互换动物名字读一遍，是否还成立？）
- [ ] emotionTag 是否和最近几个 Owner 事件连续重复（尤其是 Comfort/Concern）？这次能不能覆盖一个还没出现过的情绪？
- [ ] 感知通道是否又是"脚步声/钥匙声/响动"？能不能换成嗅觉、触觉、温度、地板震动等还没怎么用过的通道？
- [ ] perception 有没有提前把某个选择的动作/结果写死，导致玩家选另一个选项时和感知层文字自相矛盾？（Batch 1A QA 发现的真实 bug类型）
- [ ] choice.feedback 单独拿出来读、放进 Today's Memory 里读，是否依然成立？有没有代词指代不清的问题？
- [ ] 是否符合 Animal Perspective（只写动物能感知到的）？
- [ ] 是否符合 Show, Don't Explain（呈现现象，不解释"主人在做什么/为什么"）？

## 内容覆盖审查（Coverage Audit）

对已写完的 19 个事件（EAT×3 / PLY×2 / EXP×3 / OWN×2 / ENV×2 / RST×2 / SOC×2 / RAR×3）做了一次全量核查，目的是在继续补足剩下 68 个之前，先确认"一天的节奏"有没有明显的空缺、重复或失衡，而不是无差别地按分类目标数字往上凑。

### 1. 时段覆盖不均——Morning 明显偏薄

统计每个事件的 `触发时间`（含 Any）：Morning 覆盖约 10 个、Noon 约 14 个、Evening 约 13 个、Night 约 10 个。但如果只看**专属于该时段**（不含 Any）的事件：Morning 只有 3 个（EAT-001、PLY-001、EXP-003），是四个时段里最少的。清晨这个"一天的开场"时段，目前几乎全靠 Any 类事件撑着，缺少真正只属于清晨的独特情境。

**建议**：后续补充事件时，Morning 专属情境应优先分配，比如"刚睡醒时的迷糊""晨光角度带来的新鲜光斑"等只在清晨才成立的情境。

### 2. 稀有度比例与目标分布不符——当前 Common 偏少、Uncommon 偏多

19 个事件里：Common 9 个（47%）、Uncommon 6 个（32%）、Rare 3 个（16%）、Secret 1 个（5%）。对照稀有度分层表的目标（60% / 25% / 10% / 5%），Common 明显不足、Uncommon 明显偏多。这在 19 个事件的小样本里不算问题，但如果后续补充时不特意纠偏，扩到 87 个之后比例会继续跑偏——**建议后续新写的事件里，Common 稀有度应该占明显多数**，尤其是"进食""休息"这类日常类别。

### 3. 分类完成度不均——Owner 类别进度最慢，但它对叙事最关键

按已完成比例排序：Rare 3/8（38%，反而领先）> Eating 3/12（25%）> Play/Explore/Environment/Rest/Social 都在 17～20% > **Owner 2/12（17%，最慢）**。

Owner 类别是社会层（羁绊/信任/依赖）和 Today's Memory"高光时刻"最主要的内容来源（呼应 ANIMALS.md 每种动物的"结局倾向"），目前进度却是八个分类里最慢的。如果这个类别持续落后，会直接削弱回忆系统的情感浓度。**建议优先补 Owner 类别**，而不是按现有比例均匀推进。

### 4. 主题重复风险——"门口来了不熟悉的人/声音"这个情境已经出现三次

`SOC-001`（门外的陌生人）、`SOC-002`（快递到了）、`RAR-001`（家里来了客人）三个事件本质上是同一种情境的变体，情绪标签也高度重叠（Alertness / Alertness / Surprise）。19 个事件里已经占了 3 个（16%），如果同一局里连续抽到两个，体验上会显得"总在应付敲门声"。**建议**：后续不再新增"门口/陌生人"主题的事件，除非在触发方式或选择结构上明显区别于这三个（比如把陌生人换成"陌生动物"，交互逻辑完全不同）。

### 5. 内容量直接决定"跨局至少 40% 不同"能不能兑现

DESIGN.md 和 EVENTS.md 都写了"连续两局之间至少 40% 事件不同"的目标。以现在 19 个事件、单局大概能触发 15～18 个来看，两局之间几乎不可能有 40% 的差异空间——这不是抽样算法的问题，是**内容量本身还没到能支撑这个目标的规模**。这条目标要真正兑现，至少要把总量补到能让"单局触发数"明显小于"总事件数的一半"（比如总量 60+，单局仍然只触发 15～18 个）。

### 6. 情绪光谱偏向"发现新奇"，缺少"平淡/独处"类情绪

19 个事件覆盖了 15 种不同情绪标签，多样性本身不错，但整体基调偏向"好奇/期待/警觉"这类"有事发生"的情绪，缺少纯粹的"无聊""孤独""若有所思"这类更贴近 Design Motivation 里"不是每天都有大事发生"的平淡情绪。**建议**在补充 Rest、Explore 等类别时，刻意留一些"什么都没发生，但也没关系"的情绪标签方向。

### 7. 天气内容不对等——Wind 和 Snow 目前完全没有对应事件

现有 19 个事件里，`天气` 字段用到的只有 `Any`、`Sunny`（3个）、`Rain`（2个），`Wind` 和 `Snow` 一个事件都没有覆盖到。当天天气随机到刮风或下雪（合计约 30% 概率）时，天气这个机制对当天的事件筛选**没有产生任何实际影响**，玩家也感受不到"今天风很大/下雪了"这件事的存在。**建议**至少各补 1～2 个 Wind、Snow 专属事件，让天气系统真正参与体验，而不是一个只有一半选项生效的隐藏机制。

### 8. 部分前置条件在一局短体验里可能几乎摸不到

`RAR-002`（主人今天请假）要求 `Bond > 40`，而角色初始 Bond 只有 25，且没有稳定的被动增长途径，只能靠特定选择（如 OWN-001 选 A 一次性 +20）累积。再叠加它本身是 Rare 稀有度（10% 权重），在一局 15～18 次的抽样里，"先攒到 Bond>40，又刚好被抽中"的概率可能低到接近不可能。**建议**：要么确认这就是"多周目专属、单局几乎不该出现"的设计意图并在文档里明说，要么适当降低门槛（比如 Bond > 30）让它在长线游玩中真的有机会被摸到。
