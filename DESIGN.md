# 《体验小动物的一天》项目设计文档

> **版本说明**：本文档为项目重新定位后的设计文档，替代此前"养成/经营向"的旧版设计。
> 项目现在的定位是**第一人称叙事体验游戏**，而不是数值养成游戏。
> ⚠️ 当前 `index.html` 中的代码仍然是旧定位（多天无限循环的数值经营原型），**尚未按本文档重写**。本文档只做设计对齐，不涉及任何代码改动；代码何时、如何向本文档靠拢，见第11节"后续扩展路线"。

---

## Design Motivation

现代人每天都在以"人的视角"理解动物。

我们认为宠物幸福，是因为吃得好、睡得好；
我们认为流浪动物可怜，是因为没有主人。

但动物真正感受到的世界，
并不是"有没有主人"，
而是：

今天有没有闻到新的味道？

有没有晒到太阳？

有没有遇见熟悉的人？

有没有安心睡着？

《体验小动物的一天》希望通过第一人称互动，
帮助玩家暂时离开人的视角，
体验一种更加本能、更简单的生命节奏。

它不是教育玩家，
而是邀请玩家去体验另一种生命。

## Research

Design Motivation 的立意、目标用户的判断、Animal Perspective 里的物种档案，都不应该是凭空写出来的——它们需要有一条 Research 路径支撑。这一节记录的是这条路径本身，以及它如何一步步收敛成后面第1～11节的 Game Design。

### 研究流程

```
Research
   │
   ▼
观察猫行为 ──────┐
   │             │
   ▼             │ （一手观察 + 二手资料，互相校验，
动物行为资料 ──────┤   避免单次观察产生偏差或过度拟人化解读）
   │             │
   ▼             │
Pet Owner Interview
   │
   ▼
Persona
   │
   ▼
Journey Map
   │
   ▼
Game Design
```

### 各阶段说明

1. **观察动物行为（一手观察 / Primary Observation）**：直接、如实地记录真实动物的具体行为细节——不是"猫很开心"，而是"尾巴尖轻轻抖动""耳朵微微后压""身体贴地"这类具体可描述的动作。这是保证第7节事件层级里"感知层"文案不失真的关键素材来源：事件文案里出现的每一个描述性动作，理论上都应该能在观察记录里找到依据，而不是凭想象编造。

2. **动物行为资料（二手资料 / Secondary Research）**：用已有的动物行为学资料（如猫 / 狗 / 兔的行为学科普、动物行为学文献）校验、补全一手观察，避免单次观察带来的偏差或拟人化解读。这部分资料是第8节 Animal Perspective 里各物种感官/行为档案的直接依据来源；后续新增物种视角（如仓鼠、鸟类）时，也应该先补齐这一步的资料收集，再补写档案。

3. **Pet Owner Interview**：访谈真实的宠物主人，了解主人如何理解、描述自己宠物的状态与需求，尤其关注"主人以为的"与"动物实际感受到的"之间的落差。这一步正是 Design Motivation 里核心观察的来源——"我们认为宠物幸福，是因为吃得好、睡得好"，这正是典型的主人视角表达；Research 阶段要做的，就是从访谈里找到这种视角与动物真实感知之间的缝隙，作为整个项目立意的依据。

4. **Persona**：基于观察与访谈，梳理出两条并行的 Persona，而不是只做玩家一侧：
   - **玩家 Persona**（对应第2节 目标用户）：谁会想要体验这样一款游戏、为什么。
   - **动物行为侧写**（对应第8节 Animal Perspective）：每种可选动物的感官特点与行为倾向，作为写事件、设计感知层内容的依据。

5. **Journey Map**：把"一天"作为体验主线，梳理动物（以及代入后的玩家）在这一天中会经历的情绪起伏与关键节点。这是第4节 Game Loop 与第5节 UX Flow 的前身——正式设计稿里的循环结构和流程图，都是从 Journey Map 提炼出来的。

6. **Game Design**：Research 阶段的所有产出，最终收敛为第1～11节的具体设计——也就是本文档从"产品定位"开始的全部内容。

### 研究状态说明

目前 Design Motivation 与第8节 Animal Perspective 呈现的，是基于已知动物行为学常识与合理推测整理出的设计假设，**尚未经过真正的一手观察记录与 Pet Owner Interview 验证**。建议在后续开发前，真正执行一轮观察猫行为的记录、查阅动物行为学资料，并至少完成几场 Pet Owner Interview，用真实素材替换或校验当前基于常识推导的设计假设，确保 Journey Map 与 Game Design 的细节经得起推敲。

## 1. 产品定位（Product Vision）

**《体验小动物的一天》** 是一款**第一人称叙事体验游戏（First-person Narrative Experience）**，而非养成/经营模拟游戏。

- 玩家不是站在"饲养主"的上帝视角去管理一只动物的各项数值，而是**成为**这只动物本身——从第一人称视角，真实地度过它一天中的清晨到夜晚。
- 游戏想让玩家短暂地"活成"一只猫、一只狗或一只兔子：感受饥饿驱使身体行动、感受困意不由自主地袭来、感受被投喂时那种简单又直接的满足感。这是一种**换位体验/共情式**的游戏内容，而不是"把动物照顾到满分"的目标导向内容。
- 游戏不追求"赢"。它追求的是：玩家在几分钟到十几分钟的时间里，短暂抽离"人类视角"，体验一种更本能、更简单的生存节奏，并在一天结束时，得到一段专属于这次选择的"回忆"，而不是一个分数。
- 定位对标：治愈系/慢节奏的叙事体验类独立游戏，强调情绪与氛围的传达，而非系统深度或数值成长曲线。

---

## 2. 目标用户（Target User）

- **碎片时间放松者**：通勤路上、课间、工作间隙，想用几分钟获得一次轻松、无压力的沉浸体验的用户。
- **治愈系/萌宠内容爱好者**：喜欢猫狗兔子等小动物题材，享受"云吸猫云吸狗"式内容的用户。
- **叙事/共情体验偏好者**：对"换个视角看世界"感兴趣，喜欢短篇叙事、氛围化体验，而非追求数值成长、排行榜、竞技对抗的玩家。
- **轻度/非硬核玩家**：不需要学习成本、不需要重复练习技巧即可完整体验一次，适合非游戏核心受众。
- 与旧定位的差异：**不是**面向"喜欢经营模拟、喜欢管理数值把宠物养到最佳状态"的玩家群体设计的。

---

## 3. 核心体验（Core Experience）

- **第一人称沉浸叙事**：所有场景文案从动物的感官与本能出发描述（例如"你的鼻子动了动，一阵香味飘来……"），而不是旁观者式的第三人称描述。
- **选择是本能反应，不是策略决策**：玩家做出的每个选择，呈现上是"此刻我想怎么做"的直觉反应（如"扑上去""继续趴着不动"），而非"选这个能让数值更优"的资源管理决策。底层仍然有状态数据支撑（见第6节），但呈现给玩家的是感受化的文字，而不是赤裸的数值增减提示。
- **没有失败收尾**：不存在"没喂饱就游戏结束"这类惩罚性结果。任何选择都会带出不同质感的后续情节，选择本身没有对错之分。
- **完整而有限的一天**：体验有明确的起点（清晨苏醒）和终点（夜晚入睡），是一段完整的、有始有终的叙事闭环，而不是可以无限进行下去的经营循环。
- **留下的是"回忆"，不是"评分"**：一天结束后，游戏呈现的不是分数或标签，而是一段 **Today's Memory**——由当天真实发生过的具体片段拼成的第一人称回忆文字，让玩家在回顾中产生情感共鸣，而非对"养得好不好"进行评判（详见第10节）。

---

## 4. 游戏循环（Game Loop）

与旧版"数值管理循环"不同，新的核心循环是**叙事推进循环**：一天只走一次，走到底就结束。

```
┌───────────────────────────────────────────────┐
│  开场：选择"今天要作为哪种小动物"                │
│  （猫 / 狗 / 兔子 —— 即选择一种"生命视角"）      │
└───────────────────────┬─────────────────────────┘
                         │
                         ▼
              【苏醒场景】第一人称开场白
        （"晨光透过窗帘缝隙，你睁开了眼睛……"）
                         │
                         ▼
        ┌─────────────────────────────────┐
        │   时间持续推进（清晨 → 深夜）      │◄──────┐
        └───────────────┬─────────────────┘         │
                         │ 到达某个时间节点            │
                         ▼                          │
            触发一段叙事事件（感官描述 + 选择）        │
                         │                          │
                做出一个"本能反应"选择                 │
                         │                          │
        更新内在状态（饥饿感/心情/羁绊感等，见第6节）    │
        并把这次选择记录进"这一天发生了什么"的轨迹       │
                         │                          │
              回到时间推进，继续走向夜晚 ───────────────┘
                         │
                （到达设定的"一天结束"时间点，如深夜）
                         ▼
              【入睡场景】困意袭来，一天落幕
                         │
                         ▼
        根据这一天的状态与选择轨迹，匹配并展示
              【Today's Memory 回忆】（见第10节）
                         │
                         ▼
        提供"再体验一天 / 换一只动物"的入口，回到开场
```

**与当前 `index.html` 实现的关键差异**：现有代码是"多天无限循环 + 数值自然漂移 + 永不结束"的经营结构；新定位要求收敛为**单日、有始有终、以 Today's Memory 收尾**的一次性体验。这是后续开发需要调整的核心方向。

## 5. 交互流程（User Flow / UX Flow）

交互设计项目需要一张能一眼看清"玩家会经过哪些状态"的 UX Flow 图，作为所有界面设计、动画设计的总纲；具体每一步的文案与系统细节，再展开到下面的详细说明中。

### 5.1 UX Flow 总览

```
进入游戏
   │
   ▼
选择动物
   │
   ▼
身份转换动画
   │
   ▼
一天开始
   │
   ▼
┌─→ 事件 ──▶ 反馈 ──▶ 继续 ─┐      （事件→反馈→继续 反复循环，
│                          │       直到时间到达夜晚）
└──────────────────────────┘
   │
   ▼
夜晚
   │
   ▼
回忆
   │
   ▼
分享
```

`事件 → 反馈 → 继续` 是全程唯一会重复的内层循环，对应第4节游戏循环里"时间推进 → 触发事件 → 选择 → 更新状态"的主循环；一旦时间到达设定的夜晚节点，就会跳出这个循环，走向"夜晚 → 回忆 → 分享"这条不可逆的收尾线，呼应第3节"完整而有限的一天"的核心体验原则。

### 5.2 各节点说明

| UX Flow 节点 | 对应界面/行为 | 关联章节 |
|---|---|---|
| 进入游戏 | 打开页面，进入开场界面 | — |
| 选择动物 | 从猫 / 狗 / 兔子中选择一种，配一句引导语（如"此刻，你是一只……"），强化"你即将成为它"而非"你即将饲养它"的心理预期 | 第2节 目标用户、第8节 Animal Perspective |
| 身份转换动画 | 简短的"苏醒"文字/动画，建立第一人称沉浸感，让玩家从"人类视角"过渡到"动物视角" | 第3节 核心体验 |
| 一天开始 | 时间轴启动，进入清晨，主循环开始运转 | 第4节 游戏循环、第9节 时间系统 |
| 事件 | 世界层信息被动物感知，呈现为感官化的情境描述 | 第7节 事件系统（Event Layer 世界层/感知层） |
| 反馈 | 玩家做出本能选择后，动物产生情绪反应，状态随之更新，屏幕以氛围化方式呈现（光影、动物姿态、简短感受提示），弱化"进度条式"的数值管理观感 | 第6节 状态系统、第7节 事件系统（情绪层/反馈层） |
| 继续 | 时间继续推进，回到"事件"节点，如此反复，直到抵达夜晚 | 第4节 游戏循环 |
| 夜晚 | 到达设定的"一天结束"时间点，触发入睡场景，主循环结束 | 第9节 时间系统 |
| 回忆 | 展示基于当天具体事件片段拼成的 Today's Memory 回忆文本（第一人称回忆口吻） | 第10节 回忆系统（Today's Memory） |
| 分享 | 提供"保存/分享今天这段回忆"的入口，以及"再体验一天 / 换一只动物重新开始"的入口，回到"进入游戏" | 第11节 后续扩展路线 V3（回忆分享卡片） |

> **进度提示**：`分享` 节点对应的是第11节 V3 阶段规划的功能（生成可保存/分享的"我的一天"结语卡片）；在 V1 阶段，这一节点可以先只提供"再体验一天 / 换一只动物"的入口，分享能力作为长期目标逐步补齐，UX Flow 里先把完整的终态画出来，方便对齐远期方向。

## 6. 状态系统

状态数值依然存在于底层，但其角色从"玩家要优化到满的养成目标"重新定义为**驱动叙事分支与 Today's Memory 生成的隐藏燃料**。

为了让状态系统"更像动物"、也更方便持续扩展，状态不再是三个并列的独立数值（Hunger / Mood / Bond），而是采用**分层结构**——从最底层的生理本能，逐级堆叠到最上层的社会关系：

```
生理层（Physiological Layer）
  饥饿　疲劳　安全感
        │
        ▼
情绪层（Emotional Layer）
  好奇　放松　紧张
        │
        ▼
社会层（Social Layer）
  羁绊　信任　依赖
```

| 层级 | 子状态 | 说明 |
|------|--------|------|
| 生理层 | 饥饿、疲劳、安全感 | 最底层、最直接由时间和环境驱动的本能信号，动物自己无法主观控制；例如饥饿随时间自然增长，疲劳随清醒时长增长，安全感会因环境刺激（陌生声音、靠近的人）而瞬时波动 |
| 情绪层 | 好奇、放松、紧张 | 由生理层状态与"感知到的世界信息"共同触发的即时情绪反应；例如饥饿升高时听到声音更容易触发"好奇"，安全感偏低时听到陌生声音更容易触发"紧张"。这一层与第7节事件层级中的"情绪层"节点是同一个概念，是两套系统的交汇点 |
| 社会层 | 羁绊、信任、依赖 | 更长期、更缓慢积累的社会性状态，由动物与"主人"/环境反复互动的历史塑造，不由单次事件决定，而是多次选择趋势的累积；主要服务于第10节回忆系统挑选片段、决定回忆语气基调的内部参考 |

**与旧版三个数值的对应关系**：旧版的 `Hunger` 对应生理层的"饥饿"；旧版的 `Mood`（单一心情值）被拆解为情绪层的三个更细腻的维度（好奇/放松/紧张），不再是单一的好坏刻度；旧版新增的 `Bond` 对应社会层的"羁绊"，社会层同时补充了"信任""依赖"两个相关维度。

设计原则：
- **层级之间是自下而上的因果关系，而非三层各自独立打分**：生理层状态会影响情绪层情绪被触发的阈值（如饥饿越高，越容易被"食物气息"触发好奇或急躁）；情绪层的反复模式会缓慢累积影响社会层（如多次因为紧张而选择"躲起来"，会让信任缓慢下降）。
- **子状态不追求"越高越好"**：例如"紧张"不是需要被消灭的负面惩罚值，而是一种如实记录的情绪信号，游戏不会因为紧张值高就判定"失败"。
- **结构本身为未来扩展预留了空间**：新增维度只需要加进对应层级，不需要改动整体架构——例如生理层未来可加入"体温""口渴"，情绪层可加入"喜悦""厌恶"，社会层可加入"归属感"等。
- 状态**不向玩家展示"及格线"或"游戏结束阈值"**，弱化对抗感、弱化"管理不好会怎样"的焦虑；其意义在于"记录这一天是怎么过的"，而不是"证明你养得好不好"。
- 呈现方式应优先考虑氛围化表达（动物姿态、光影、文字语气），数值条可以保留作为辅助信息，但不作为核心反馈手段。

## 7. 事件系统

事件从旧版"选项 + 数值对照表"的经营逻辑，升级为**事件层级（Event Layer）**模型——这是本次事件系统重新设计的核心。

### 7.1 事件层级（Event Layer）：世界与动物是分开的两层

旧版设计里，"事件"是一个整体：系统直接弹出一个命名好的事件（如"吃饭事件"），玩家从标题就已经知道"这是什么事"。

新版设计把"事件"拆成五个层级，一层层因果推导下去，玩家**只活在感知层里**：

```
世界层：世界一直在发生事情（不管动物有没有注意到，事情照样发生）
              │
              ▼
感知层：动物感知到了什么（受物种感官/所处状态影响，只是"听见/闻到/瞥见"，
        不知道背后真正发生了什么）
              │
              ▼
选择层：玩家（=动物）基于"感知到的碎片信息"做出本能选择
              │
              ▼
情绪层：这个选择让动物产生了具体的情绪/生理反应
              │
              ▼
反馈层：世界因为动物的反应而发生变化（真相在这一刻才被揭晓，
        或者动物依然不知道发生了什么）
```

**关键设计原则**：玩家在选择层看到的**永远不是"早餐事件"这样的人类概念标签**，而是"听见了塑料袋的声音"这种纯粹的感知信息。动物不懂"早餐""主人准备食物"这些概念，它只懂"有声音""有味道""这个人我认识"。这正是 Design Motivation 里想传达的——离开人的视角，回到动物真正的感知方式。

### 7.2 示例：厨房传来的声音

| 层级 | 内容 |
|------|------|
| 世界层 | 07:00，主人在厨房打开了一个塑料袋（准备早餐）——这件事无论动物是否注意到都会发生 |
| 感知层 | 猫竖起耳朵：**"厨房传来一阵塑料袋的声音……"**（呈现给玩家的仅此而已，不告诉玩家"这是主人在准备早餐"） |
| 选择层 | A. 去看看　B. 不理　C. 躲起来 |
| 情绪层 + 反馈层（按选择分支） | **A 去看看**：走近后发现真相——可能是食物袋，被投喂，情绪走向满足/好奇被验证；也可能只是空袋子，情绪走向轻微失落。<br>**B 不理**：继续待在原地，声音渐渐停止，动物始终不知道刚才发生了什么，情绪平静但饥饿感持续累积。<br>**C 躲起来**：出于警惕本能回避，安全感短暂上升，但同样错过了确认真相的机会，事后可能因错过进食而更饿。 |

同一个"厨房声音"的世界层事实，会因为选择层的不同分支，让动物走向完全不同的情绪体验和世界反馈——**结果不同，但没有对错之分**。

### 7.3 与旧版事件系统的关系

- **固定事件与随机事件的结构仍然保留**，但重新理解为"世界层"里既定或随机发生的世界状态变化（如"厨房有动静""窗外有声响""门口有脚步声"），动物层面统一通过"感知层"的方式呈现给玩家，复用现有的按游戏内时间调度、避免弹窗重叠的技术机制。
- **文案与呈现规则**：
  - 感知层文案只描述"感官信息"（听见/闻到/看见/触到），不描述"事件本质"，事件本质留到反馈层才可能揭晓（也可能永远不揭晓）。
  - 选项文案强调"本能行为"（"扑上去""继续观察""往声音的方向走去""躲起来"），而不是"策略选择"式的表达。
  - 情绪层与反馈层的结果，除了更新状态数值外，还需要**记录进"这一天发生了什么"的轨迹**（例如：今天选择了几次"靠近确认"类选项、几次"回避"类选项），供第10节回忆系统在日终时读取、拼接成 Today's Memory。
- 事件之间不再单纯以"数值是否达标"作为唯一调度依据，还可以结合时间段、已发生的事件类型、动物当前的警觉/信任状态，让当天的情节走向更有叙事连贯性（例如同一天内不应重复触发完全相同的情境两次）。

### 7.4 Event Library

事件*内容*（具体的 87 个事件条目、字段规格、稀有度分层、Event ID 命名规则）已独立成单独文档 **[EVENTS.md](./EVENTS.md)**，本节只保留和"系统"相关的概要，避免重复维护。

- **规模**：总事件数量 80～100 个（当前规格定为 87 个），分 8 类：🍽 进食(12) / 🏠 主人互动(12) / 🌿 探索(15) / 😴 休息(10) / 🎈 玩耍(10) / 🌧 环境(10) / 🐾 社交(10) / ✨ 稀有事件(8)。
- **单局抽样规则**：每次游玩从事件库中随机抽取约 30～40 个参与调度（对应第9节"约 40 场事件 / 8～12 分钟"的节奏目标）；同一局不重复；连续两局之间至少 40% 事件不同；稀有度更高的事件权重更低（见 EVENTS.md 的稀有度分层）。
- **稀有度与情绪标签**：EVENTS.md 为每个事件额外定义了「稀有度」（Common/Uncommon/Rare/Secret，与"分类"是独立的两个轴）和「情绪标签」（Curiosity/Fear/Relief 等），其中情绪标签会反过来影响第10节 Today's Memory 的生成方式（见第10.3节的更新）。
- 抽样时仍然遵循本节前文的调度规则（不在同一天重复触发完全相同的情境、结合时间段与动物当前状态），事件库只是扩大了"可供调度的候选池"，调度逻辑本身不变。
- **当前 `index.html` 里只有 5 个事件**（morning / toy / window / sunspot / ownerHome），是 EVENTS.md 里 87 个目标规模事件的一个早期子集，尚未按 EVENTS.md 的完整字段规格（稀有度/适用动物星级/触发时间/天气/前置条件/情绪标签）重写。

## 8. Animal Perspective（动物视角档案）

第7节事件层级里提到，"感知层"能感知到什么，取决于动物的物种感官与行为特点——这一节就是为每种可选动物建立具体的**感知与行为档案**，作为写事件、写感知层文案时的依据。更详细的可执行规格（性格关键词、感官星级、专属事件清单、Idle 动作、Today's Memory 语气倾向）见独立文档 **[ANIMALS.md](./ANIMALS.md)**。

**设计目的**：有了这份档案，写事件时就不会写成"猫：今天去散步吧"这种人类安排式的句子，而是写成"窗外飞过一只鸟，你盯住了它"——后者才是动物真实的、被动的行为反应。

### 猫
- 视觉：对动态高度敏感，夜视能力优秀，辨色能力较弱
- 听觉：极其敏锐
- 嗅觉：很强
- 行为特点：白天大部分时间在睡觉，喜欢观察，性格谨慎

### 狗
- 嗅觉：极强，是最主要的感知方式
- 行为特点：更依赖主人，更喜欢互动，容易兴奋

### 兔子
- 行为特点：天性胆小，容易受惊，喜欢躲藏，需要持续的安全感

### 设计应用
- **同一个世界层事实，落到不同动物身上要有不同的感知层反应**：例如"窗外飞过一只鸟"——猫会"盯住它，尾巴尖轻轻抖动"（视觉动态敏感 + 谨慎观察）；狗可能"根本没注意到，鼻子还埋在地上找味道"（更依赖嗅觉而非视觉）；兔子可能"整个身体瞬间僵住，耳朵转向声音的方向"（胆小易惊）。这份档案就是判断"这只动物此刻会注意到什么、会怎么反应"的依据。
- **第6节状态系统里的层级阈值也应因物种而异**：例如兔子的"安全感"基线更低、更容易触发"紧张"；狗的"社会层·依赖"更容易随互动累积上升；猫的"情绪层·好奇"更容易被动态视觉线索触发。这些差异暂不在本文档里逐一列出具体数值，留待原型阶段调参。
- 这份档案与第7节事件层级、第6节状态系统、以及第11节 V2 阶段"为猫/狗/兔子设计差异化生活节奏事件池"共同构成同一套底层依据；未来新增物种（如仓鼠、鸟类）时，只需要补充一份类似结构的档案。

## 9. 时间系统

- 延续现有的时间流速比例：**现实 1 秒 = 游戏内 1 分钟**（现实 1 分钟 = 游戏内 1 小时），即一整天（24 小时）约对应现实 24 分钟；后续可根据"短体验"定位适当压缩（例如让一天对应现实 10～15 分钟），具体数值留待原型测试后确定。
- **与旧版的关键差异**：时间不再无限跨天滚动，而是拥有明确的**起点（清晨苏醒）与终点（入夜入睡）**，构成一个有始有终的单日体验闭环，而不是可以无限游玩下去的经营系统。
- 时间除了驱动事件调度外，还承担**叙事节奏**的作用：清晨、正午、黄昏、深夜等不同时段应对应不同的光影氛围与事件基调，强化"陪伴一只动物走完一天"的完整感和仪式感。

## 10. 回忆系统（Today's Memory）

**这是本次重新定位新增的系统，当前 `index.html` 尚未实现。**

### 10.1 为什么不是 "Ending"，而是 "Today's Memory"

"结局（Ending）"这个词本身就带着"好结局 / 坏结局"的评判语气——它暗示这一天是一场需要被打分的表现。这与 Design Motivation 里"不教育、不评判，只是邀请玩家体验另一种生命"的立场是矛盾的。

所以这一节不再叫"结局系统"，而是 **Today's Memory（今日回忆）**：一天结束后留下的，不是一个被归类、被打分的结果标签，而是一段具体的、朴素的回忆片段——就像动物自己也不会用"我今天过得好不好"去总结一天，只会留下一些具体的画面：晒到的太阳、蹭过的腿、听过的声音。

### 10.2 触发时机

游戏内时间到达"一天结束"的设定节点（如深夜、动物明显困倦），触发"入睡"场景，随后进入 Today's Memory。

### 10.3 生成逻辑：拼接真实发生过的片段，而不是匹配一个标签

不再是从几个预设好的"结局类型"里挑一个标签展示给玩家，而是：

- 从第7节提到的"这一天发生了什么"的事件轨迹里，**挑选出几个真实发生过、有代表性的具体片段**（例如"大部分时间趴在窗边""听到厨房的声音选择了去看看""主人回来时慢慢走过去蹭了蹭腿"）。
- 用平实、具体、第一人称回忆口吻的短句呈现这些片段，只做如实描述，不做归纳评判（不说"你是一只热情的猫"，只说"你走过去，蹭了一下他的腿"）。
- 第6节状态系统（尤其是社会层的羁绊/信任/依赖）不再用来"匹配一个结局类型"，而是作为**挑选哪些片段更值得被记住、决定回忆整体语气基调**的内部参考——这个基调不会以标签形式呈现给玩家，玩家只会看到最终流畅的一段回忆文字。

**情绪标签统计（与 [EVENTS.md](./EVENTS.md) 联动）**：[EVENTS.md](./EVENTS.md) 里每个事件都定义了一个「情绪标签」（如 Curiosity / Relief / Fear）。生成 Today's Memory 时，在挑选具体片段之外，再做两步统计，让回忆不只是"挑事件"，也"读情绪"：

- **高频情绪**：统计当天所有已发生事件的情绪标签，找出出现频次最高的一个，生成整体基调句，如"今天，你一直在好奇这个世界。"
- **高光时刻**：从当天事件里挑一个"独一无二"的情绪时刻（例如全天唯一一次 Relief），生成单点句，如"今天，你最安心的时刻，是听见主人回家的脚步声。"

这两句可以和上面"挑选真实片段"的逻辑拼在一起：前者负责挑"发生了什么"，情绪标签负责挑"感觉像什么"，两者结合，Today's Memory 才不只是数值判断，而是真正像是在回忆这一天。

### 10.4 示例（Today's Memory 文案参考）

> 今天，
> 你大部分时间都趴在窗边。
> 太阳很好。
>
> 主人回来时，
> 你没有马上跑过去。
> 只是慢慢走过去，
> 轻轻蹭了一下他的腿。
>
> 今天，
> 也是很好的一天。

### 10.5 呈现形式与设计原则

- **呈现形式**：一段简短的回忆文字，以第一人称回忆口吻呈现，可搭配一张静态氛围插画或场景收尾；**不出现分数、星级、标签、排行榜等任何评判性 UI 元素**。
- **不存在"更好的回忆"或"更差的回忆"**：每一天的 Today's Memory 都只是"你恰好经历的这一天"，具体、独特、不可复刻，不分高低优劣。
- **收尾句式保持一种温柔的、无差别的肯定语气**（如示例里的"今天，也是很好的一天"）——同一种结局类型内，无论这一天发生了什么、玩家做出了怎样的选择，这句收尾都不因此改变。这是刻意的设计，用来消解"我是不是选错了"的评判焦虑，呼应 Design Motivation 的立场。**（第13节新增的例外）**：这条"不变"仅在同一个 Ending 类型内部成立；不同 Ending 类型（如 Normal / Death）各自拥有自己的一句固定收尾，跨类型允许不同，但每种类型内部依然是无差别的——避免"物种/选择差异"这条早已放弃的方向借着 Ending 类型的引入重新变相回来。
- **内容应优先引用当天真实发生过的具体事件片段**，而不是笼统的状态总结，这样每一次体验产出的回忆才会因为玩家的选择而真正不同，也让回忆读起来像"发生过的事"，而不是"系统生成的评语"。

### 10.6 记忆风格模板体系（Content Freeze 阶段新增，尚未实现）

第10.3节的生成逻辑（第一片段/末片段 + 高频情绪 + 高光时刻）目前只有**一种固定结构**：不管这一天实际发生了什么，拼接方式都是同一套模板。这一节设计一组"记忆风格模板"，让 Today's Memory 的**结构**也能因为这一天的实际形状而不同，而不只是**内容**不同。

**重要边界（不能违反）**：第10.5节"收尾句式保持一种温柔的、无差别的肯定语气"这条原则不受本节影响——**所有模板的收尾都必须是同一句固定的肯定语气**，模板之间只有正文结构不同，收尾永远统一。这是为了避免"物种基调差异"和"收尾必须无差别"这两条原则打架。

| 模板 | 触发倾向（具体条件） | 结构 | 适合的物种基调 |
|---|---|---|---|
| **平静流水型**（默认） | 没有触发其他任何模板条件时的兜底模板 | 按时间顺序简单陈述 1～2 个具体片段 | 三种动物通用，是最常见的一种 |
| **情绪主导型** | 当天某个情绪标签的出现次数占比明显偏高（例如占当天全部事件的 40% 以上） | 开场即用"你一直在{情绪}这个世界"定调，其余片段作为支撑 | 猫（好奇主导时最典型） |
| **高光单点型** | 当天出现过 Rare 或 Secret 稀有度的事件 | 整段回忆围绕这一个稀有片段展开，其余内容让位，例如"今天，最忘不了的是……" | 三种动物通用，但触发本身天然稀有 |
| **社会羁绊型** | 当天 Owner 类事件发生 2 次以上，或社会层数值（羁绊/信任）比清晨上升明显 | 以主人相关的片段为主线，语气更贴近"陪伴" | 狗（"热闹相伴"最典型），信任建立后的兔子也适用 |
| **警觉起伏型** | 当天存在一次"紧张类情绪"（如 Fear/Unease/Alertness）之后又出现"平静类情绪"（如 Relief/Comfort/Calm）的先后顺序 | 先讲紧张的部分，再讲后来怎么平静下来，呈现一个小的"起伏后归于平静"的弧线 | 猫（"有惊无险"）、兔子（"紧张到放松"是 ANIMALS.md 明确列出的记忆代表行为） |
| **平淡无事型** | 当天触发的事件数量很少（比如个位数），没有明显的稀有或强烈情绪片段 | 不刻意制造起伏，如实说"今天大部分时间很安静"，收尾依然是那句肯定语气 | 三种动物通用，用来兜住"内容库还不够丰富""这一天恰好很平淡"这两种情况，避免因为事件太少而显得回忆很空洞 |

**选择顺序建议**：几个模板的触发条件可能同时成立（比如"高光单点型"的稀有事件恰好也是"社会羁绊型"想要的 Owner 事件），建议按表格从上到下的优先级取第一个满足条件的模板，而不是同时套用多个模板——保持每一天的回忆只有一种主结构，不要把所有触发条件都塞进同一段文字里。

**一个尚未解决、需要在实现代码时补的数据缺口**：要判断"高光单点型"的触发条件（当天是否出现过 Rare/Secret 事件），需要在记录"这一天发生了什么"的轨迹时，除了情绪标签之外，也记下这个事件的稀有度——目前的实现只记录了事件名称、反馈文案和情绪标签，没有记录稀有度。这个字段补充留到 Content Freeze 结束、正式实现代码时再处理，本节不涉及代码改动。

## 11. 后续扩展路线（V1～V3）

### Version 1：收敛现有原型为"单日叙事体验"
- 将现有 `index.html` 的"多天无限循环 + 数值自然漂移 + 永不结束"结构，改造为"清晨 → 夜晚"的单日闭环（有明确开始与结束）。
- 重写事件文案为第一人称感官叙事风格（保留现有事件调度机制作为技术基础）。
- 按第7.4节 / [EVENTS.md](./EVENTS.md) 规格，把事件从现有的 5 个手写事件扩充为 80～100 个、分 8 类的事件库，实现单局随机抽取 30～40 个、同局不重复、跨局至少 40% 不同、稀有度低概率出现的抽样规则，并接入情绪标签统计以丰富 Today's Memory。
- 新增"回忆系统（Today's Memory）"的最小实现：基于当天记录的具体事件片段，拼接生成一段第一人称回忆式短文，替代分数/标签式的展示。
- 弱化数值条的视觉权重，改用更氛围化的状态提示方式（不删除底层数值机制，只调整呈现）。

### Version 2：深化叙事与代入感
- 为猫 / 狗 / 兔子分别设计**有差异化生活节奏**的事件池（三种动物的"一天"在结构上应有所不同，而不只是换一个 emoji 皮肤），让每种动物能拼出的 Today's Memory 片段也各不相同。
- 把"羁绊、信任、依赖"等社会层维度落地到具体事件里，让可供 Today's Memory 挑选、拼接的片段更丰富多样。
- 加入光影变化、氛围音效等感官层，进一步强化第一人称沉浸感。
- 加入更细粒度的"日记/事件片段"过程记录，供 Today's Memory 环节引用当天具体发生过的情境，让每一段回忆更"个人化"、更不可复刻。

### Version 3：内容与体验广度扩展
- 扩展为"多种基调的一天"体验合集（如"暴雨天""生病的一天""搬新家的第一天"等特殊主题日），每次体验仍保持"单次=一个完整片段"的结构，而非无限连续经营。
- 引入更多物种视角（如仓鼠、鸟类等），体验不同物种在感官与生活节奏上的差异。
- 支持回忆分享：生成一张可保存/分享的 Today's Memory 卡片。
- 探索"不同回忆片段"的多样化体验（鼓励玩家想看看"今天又会留下怎样的回忆"，而非追求某个"最优回忆"）。

## 12. Narrative Time & Perspective Writing Guideline（文案写作规范）

这一节不是针对某几个具体案例的修补，而是**所有新增文案（事件 perception/choices/feedback、IdleNarrative、Today's Memory 模板片段）以后必须遵守的长期写作标准**。它的起因是一次playtest发现的具体问题——时段较宽（Morning/Noon/Evening/Night 四段，每段跨度4～6小时）导致部分文案用了"半夜""一整晚"这类只在时段某一端才成立的措辞，玩家在时段中段读到就会出戏——但规范本身面向的是未来所有内容产出，不只是这一类问题。

### 12.1 核心原则

1. **描述正在发生的变化，而不是已经确定的时刻。** 时段是一个跨度，不是一个点；过程性描述（"光线在变暗"）在整个跨度内都成立，时刻性断言（"现在是半夜"）只在某一瞬间成立。
2. **使用相对时间语言，不用绝对报时词。** 相对语言描述"相对于此刻的变化趋势"，不依赖玩家或系统知道确切几点；绝对时间词是在替一个本来就模糊的时段系统做精确断言，两者天然冲突。
3. **锚定当前体验，不回顾已完成的时长，也不预告未来。** 只写"此刻感受到了什么"，不写"这段时间总共怎么样"（如"一整晚"），也不提前断言"马上要发生什么"。
4. **一句文案必须在所属时段的第一分钟到最后一分钟都成立。** 这是可执行的自检标准：写完后设想这句话分别在该时段刚开始和快结束时被读到，两种情况都读得通才算合格。
5. **内容文案不直接使用 UI 层的六段时间词**：清晨 / 上午 / 正午 / 下午 / 傍晚 / 夜晚。这六个词属于顶部栏时钟显示专用的细分标签（`currentPhaseLabel()`），边界比驱动事件/Idle筛选的四段系统（`currentTimePeriod()`：Morning/Noon/Evening/Night）更细，两套系统边界并不对齐；内容层文案借用这套词汇，会在两套系统边界不重合的区间产生"文案说的和顶部栏显示的对不上"的违和感。内容层请一律改用环境变化描述。
6. **Animal Perspective（动物视角原则）**：文案优先描述动物能够直接感知的内容——声音、气味、光线、温度、身体反应、本能冲动，而不是人类才具备的社会性/日历性概念（周末、请假、上班、下班、加班、快递、放假、生日）。动物不理解"主人今天请假"这件事本身，它只能感知到"那个人一直没有离开"这个现象。这条原则和第7节 Event Layer 的设计是同一个精神的延伸：人类概念属于世界层，不该泄漏到感知层。
7. **Show, Don't Explain（描述感知，不解释原因）**：优先呈现"现象"，不要替玩家把"世界发生了什么"直接讲出来。
   - ✓ 门口传来熟悉的脚步声　✗ 主人回来了
   - ✓ 那个人一直没有离开　　✗ 主人今天请假了
   - ✓ 空气里多了一点陌生的甜味　✗ 主人买了蛋糕

   原则6和原则7经常同时命中同一句话——人类概念词（请假/下班/蛋糕）往往就是"解释"而非"感知"，去掉解释、换成动物能感知的现象，两条原则会一起被满足。

### 12.2 各时段的具体示例（以 `currentTimePeriod()` 的真实边界为准）

| 时段 | 范围 | ✓ 推荐 | ✗ 禁止 |
|---|---|---|---|
| Morning | 06:00-10:59 | 阳光刚照进来、一天开始了、身体还没完全醒 | 半夜、傍晚、一整天 |
| Noon | 11:00-14:59 | 光线很足、正是最亮的时候、慵懒的午后感 | 清晨、夜色、半夜 |
| Evening | 15:00-19:59 | 天色渐渐暗下来、影子越来越长、光线开始变了 | 一整晚、一晚上、半夜、夜深了 |
| Night | 20:00-23:59 | 天黑下来了、屋子安静下来、夜晚的气息 | 半夜、凌晨 |

### 12.3 推荐表达方式（按感知通道，给思路不给固定词表）

不整理成固定词库——如果每条新文案都从下面几个短语里挑一个组合，几十条 Idle/事件文案最终会收敛成同一批句子的排列组合。正确的用法是：写新文案前，先问**"此刻，这只动物的感官会捕捉到什么"**，再从下面某个通道展开。

| 感知通道 | 写作思路 |
|---|---|
| 听觉 | 描述声音本身、声音的变化或缺席（由远及近、突然安静） |
| 嗅觉 | 描述气味的陌生/熟悉/浓淡变化 |
| 光线/视觉 | 描述光的角度、强弱、影子的变化 |
| 身体反应 | 描述身体自发的生理/本能反应，而不是情绪标签本身（"耳朵不自觉地立起来"而不是"你很紧张"） |
| 动作冲动 | 描述"想做什么"的冲动本身，而不是解释动机（"很想凑过去看看"而不是"你很好奇发生了什么"） |

### 12.4 禁止 / 慎用词汇

| 类型 | 示例 | 原因 |
|---|---|---|
| 绝对报时词 | 半夜、凌晨、正午、子夜、拂晓、黎明 | 断言精确时刻，与宽跨度的时段桶天然冲突 |
| 完成时长断言 | 一整天、一整晚、一下午、一上午、整晚、整天 | 只在时段结束瞬间成立，时段中前段读到即出戏 |
| UI 六段时间词（挪用进内容文案时） | 清晨、上午、正午、下午、傍晚、夜晚 | 属于顶部栏显示系统，边界与内容筛选系统不对齐 |
| 未来预告（无绑定判定时） | 马上就要天黑了、再过一会就…… | 除非有机制保证只在时段快结束时触发，否则可能提前出现 |
| 人类社会性/日历概念 | 周末、请假、上班、下班、加班、快递、生日、放假 | 动物无法感知这些概念本身，只能感知其外显现象 |
| 直接解释因果 | "因为主人……所以……"、"原来是……" | 剥夺了玩家自己感受/推测的空间，也违反第7.1节"感知层不该包含世界层信息"的设计 |

### 12.5 写作自检清单

新增任何 perception / choices / feedback / IdleNarrative / Today's Memory 片段前，逐条检查：

1. 这句话如果在所属时段**刚开始**的那一刻被读到，成立吗？在**快结束**时读到，成立吗？
2. 是否用了绝对报时词或完成时长断言？
3. 是否借用了 UI 六段时间词（清晨/上午/正午/下午/傍晚/夜晚）？
4. 这句话描述的是动物能感知到的现象，还是人类才懂的概念？
5. 这句话是在"呈现现象"，还是在"直接解释世界发生了什么"？
6. 如果这句话会被记录进"这一天发生了什么"的轨迹（进而被 Today's Memory 二次引用），脱离原始触发时刻单独读，是否依然成立？

## 13. Ending Framework（结局系统）

### 13.1 概念分层：Ending Framework 围绕"结局类型"设计，不是围绕"死亡"设计

一天的结局分两层概念：

```
Ending Framework
  ├─ Normal Ending  —— 正常度过一天，到达午夜
  └─ Alternate Ending —— 非正常结束
        └─ Death（当前唯一的具体实现）
```

`Death` 只是 **Alternate Ending 的第一个实现**，不是 Ending Framework 本身。未来的 `Happy` / `Secret` / `True` 等结局，都是在 Alternate Ending（或者需要时新增的其它分类）下追加新的具体实现，而不是给 Ending Framework 重新设计一套流程——第13.2节的流水线、`ENDING_CONFIG` 的配置方式、dayLog 的统一记录方式（13.3节），对所有结局类型都通用。

更准确地说，这里其实是两个层级，不要混为一谈：

```
Ending Framework
        │
        ▼
   Ending Type（结局属于哪一大类，决定走哪条 ENDING_CONFIG 配置）
     ├─ Normal
     └─ Alternate
              │
              ▼
        Ending Variant（这一类结局具体是哪一条内容，对应 DEATH_ENDINGS 里的一条记录）
          ├─ Death-001（猫）
          ├─ Death-002（狗）
          ├─ Death-003（兔）
          └─（未来）Happy-001 / Secret-001 / True-001 ……
```

- **Ending Type** 决定"走哪条收尾流程/哪条 `ENDING_CONFIG` 配置"（余韵时长、回忆引导语、收尾句）——目前是 `Normal` / `Death`，`Death` 在这一层的语义其实是"Alternate 类型下当前唯一的一种"。
- **Ending Variant** 决定"具体呈现哪一句感知文案、哪个 id"——目前 `DEATH_ENDINGS` 数组里的三条记录就是三个 Variant，共用同一个 `Death` Type。

这只是一个文档层面的抽象说明，**不要求现在实现新的 Type**；它的作用是明确以后新增任何结局内容时该往哪一层加：同一大类下追加新的按物种/按条件区分的文案，属于新增 Variant（加一条 `DEATH_ENDINGS` 记录即可）；引入一种全新性质的结局（如 `Happy`），属于新增 Type（需要在 `ENDING_CONFIG` 里加一条配置）。

死亡不是失败，也不是惩罚，而是"体验动物一天"的一种自然结局：**有些动物，并没有活到今天结束。** 整个系统必须保持现有的叙事风格——安静、克制，不突然跳出评判性的结算画面，不血腥、不恐怖，更像一段体验的自然收束，而不是"游戏失败"。

**硬性文案约束**：玩家可见文本中，任何时候都不出现"死亡""Death""你死了"这类直接字样。只呈现动物能感知到的现象（遵守第12节 Animal Perspective / Show, Don't Explain），"发生了什么"留给玩家自己理解。这条约束比第12节的其它规则更严格，专门适用于本节涉及的所有文案。

### 13.2 架构原则：Ending 是一个类型，不是一套单独的流程

一天的结束只有一个出口，不为任何具体结局复制一套单独的收尾逻辑。所有结局类型共用同一条流水线：

```
触发条件达成（到达午夜 / Alternate Ending 判定命中）
        │
        ▼
beginEnding(endingType, leadThought, endingId)
  —— 冻结时间与交互，把这次结局作为一条统一记录写进 dayLog（见13.3），
     显示（可选的）最后一句感知文案，画面渐渐静下来（余韵阶段，见13.5）
        │
        ▼
finishDay(endingType)
  —— 按 ENDING_CONFIG[endingType] 切换 Today's Memory 的引导语和收尾句，
     调用 buildMemory() 生成正文
        │
        ▼
Today's Memory 回忆屏幕
```

新增一种结局类型，只需要在 `ENDING_CONFIG` 里加一条配置（余韵时长 / 回忆引导语 / 收尾句），不需要新增新的收尾函数或屏幕。

### 13.3 统一数据流：Ending 也是 dayLog 的一条记录

`dayLog` 之前只记录"这一天发生过的选择"（`{ name, memory, emotionTag }`）。现在 `beginEnding()` 会额外追加一条结局记录：

```js
dayLog.push({
  type: 'ending',
  ending: endingType,
  id: endingId || null,
  text: leadThought || null,
  gameMinute: totalMinutes,
  realTimestamp: Date.now()
});
```

- `type: 'ending'` 把它和普通事件片段区分开。
- `ending` 是 Ending Type（`'Normal'` / `'Death'` / 未来的其它类型）。
- `id` 是具体的 Ending Variant id（Death 目前对应 `DEATH_ENDINGS` 里的条目，如 `'DEATH-002'`；`Normal` 没有对应内容池，恒为 `null`）。
- `text` 是玩家看到的最后一句感知文案（`Normal` 目前不显示这句，恒为 `null`）。
- `gameMinute` 是结局发生时的游戏内分钟数（`totalMinutes`）——记录"这一天在第几分钟结束"。
- `realTimestamp` 是结局发生时的真实时间戳（`Date.now()`）——记录"玩家真实游玩到什么时候"。

`gameMinute` / `realTimestamp` 目前**只是记录，不参与任何判定或文案逻辑**——是给 Timeline / Calendar / Replay / Statistics / Achievement 这类以后可能会读 dayLog 的系统预留的字段（例如"结局发生在当天第几分钟""平均在什么时间结束一天"），不代表现在就要实现这些系统。

**目的不是为了 Today's Memory**，而是让"这一天怎么结束的"和"这一天发生过什么"落在同一份数据里。以后要做 Timeline / Calendar / Statistics / Replay / Achievement 之类的系统，只需要遍历 `dayLog`，不需要单独判断 `dayEndingType`。

`buildMemory()`（第10.3节）目前选择**忽略**这条记录——只用 `dayLog.filter(e => e.type !== 'ending')` 挑出真正的事件片段来拼正文，保证输出和引入这条统一记录之前完全一致。以后如果想让回忆正文也读取结局信息，直接在 `buildMemory()` 里读 `dayLog` 最后一条 `type==='ending'` 的记录即可，不需要再改数据结构。

### 13.4 Alternate Ending 的触发：独立于 EventManager 之外

Alternate Ending（当前的 Death）不是一个事件，不进入 `EVENTS_LIBRARY`，不经过 `EventManager.pickNext()` 的筛选/加权/抽样逻辑——它没有 `choices`，因为这一刻动物没有选择的余地（这正是它和普通 Event Layer 的核心区别：普通事件代表"动物的本能反应/能动性"，Alternate Ending 代表"这一刻没有能动性"）。

内容池 `DEATH_ENDINGS` 是数组（不是按动物的字典），每条记录带 `id / animal / perception` 三个字段：

```js
{ id: 'DEATH-002', animal: 'dog', perception: '……' }
```

`id` 现在只在 dayLog 里被原样记录，暂不消费，是给 Memory / Statistics / Secret Ending / Replay 这类以后可能会读 dayLog 的系统预留的字段；数组结构则是为了以后给同一种动物追加多条内容时，不需要改数据形状（当前每种动物只有一条）。

判定挂在和普通事件相同的检查节奏上（每次到点检查要不要出事件时，先掷一次），所有相关数字集中在 `DEATH_CONFIG` 里，调参只改这一处：

```js
const DEATH_CONFIG = {
  baseChance: 0.0015,           // 每次事件检查节点的基础概率
  graceMinutes: 60,             // 苏醒后的宽限游戏分钟数，此期间内不判定
  safetyThreshold: 25,          // 安全感低于此值，触发轻微加成
  safetyBonus: 0.2,
  tensionThreshold: 70,         // 紧张值高于此值，触发轻微加成
  tensionBonus: 0.15,
  badWeather: ['Rain', 'Wind'], // 触发轻微加成的天气类型
  weatherBonus: 0.15,
  multiplierCap: 1.5            // 以上加成叠加后的总封顶，保证影响始终"轻微"
};
```

- **极低基础概率**，且苏醒后的第一个游戏小时内不判定（避免"刚睁眼就结束"的突兀感）。
- **允许受到轻微的环境/状态影响**（安全感很低、紧张值很高、恶劣天气），但每项加成很小且总和封顶，确保玩家不会把它归因成"这次选错了"——它始终更接近"意外"，而不是"后果"。
- 三种动物各有一条独立的感知文案（猫：好奇驱使的靠近；狗：因为信任而靠近；兔子：惊慌逃跑中撞入危险），只描述感官现象，不解释原因。

### 13.5 余韵阶段（Afterglow）

结局触发后不立刻切换到 Today's Memory：画面先停住（动物停止浮动、光晕与状态指示渐隐），最后一句感知文案停留片刻，几秒后再真正收尾。`Normal` 结局沿用原有的即时节奏（余韵时长为 0，行为和引入本框架之前完全一致）；`Death` 结局有一段约 3～5 秒的停留。这个时长由 `ENDING_CONFIG` 按结局类型各自配置。

### 13.6 Today's Memory 的结局分支

`buildMemory()` 挑选/拼接具体片段的逻辑（第10.3节）不区分结局类型——死亡发生前动物真实经历过的选择，依然正常出现在回忆里（见13.3节的 `realEntries` 过滤）。唯一的区别是引导语和收尾句：

- **正文逻辑完全复用**：第一/最后片段、高频情绪、高光时刻等既有拼接规则不变。
- **仅替换收尾句**（第10.5节的例外，见该节标注）：`Normal` 保持"今天，也是很好的一天。"；`Death` 换成同样克制、不点名"死亡"的一句（如"最后，世界安静了下来。"）。
- 每种结局类型各自的收尾句都是**固定不变**的——内部依然遵守"无差别肯定/克制语气"的原则，只是不同结局类型之间不再共用同一句。

### 13.7 开场隐藏玩法："不睁开"

身份转换过渡界面新增第二个按钮"不睁开"，与 Ending Framework 是两套独立的东西——这一刻游戏状态尚未开始（没有 `state`/`dayLog`），不需要经过 `beginEnding`/`finishDay`。点击"不睁开"后在当前页面追加一句文案（猫/狗/兔各自独立撰写，呼应各自在苏醒文案里的感官细节），不进入动物体验；玩家可以随时点击"睁开眼睛"让这句文案消失、正常开始这一天。

### 13.8 后续扩展（暂不实现）

- 死亡事件按物种扩充为多条，降低稀有内容的重复感（`DEATH_ENDINGS` 已经是数组，直接追加即可）。
- 引入更多 Alternate Ending 具体实现（`Happy` / `Secret` / `True` 等），验证 `ENDING_CONFIG` 与 dayLog 统一记录方式的扩展性。
- 结局收尾句按物种/按具体触发内容进一步分化（当前是每种结局类型共用一句）。
- 消费 `dayLog` 里的 `type==='ending'` 记录，构建 Timeline / Statistics / Replay 等新系统。

### 13.9 Future Ideas（仅记录设计想法，不在本次实现）

**Death 前的 Foreshadow（预兆）**：在真正进入 Alternate Ending 之前，先插入一句极短的"预兆"感知文案，下一次检查才真正触发结局，让死亡看起来更像"世界自然发生了一件事"，而不是"程序突然随机结束"。例如：

- 兔子："草丛突然安静下来。"
- 猫："身上的毛忽然全部竖了起来。"
- 狗："空气里多了一点陌生的味道。"

## 14. Event Design Methodology（事件设计方法论）

这一节不是某次 QA 的记录，而是把 Content QA v2（Batch 1 Review）里的发现，沉淀成**以后所有事件都要遵守的设计方法**。核心动机：QA v2 发现的最大问题不是"某个题材写多了"，而是"玩家在反复做同一种决策"——同一件事发生在门口/窗外/风里，故事各不相同，但玩家那一分钟其实在玩同一个小游戏（去看看/不去看看）。这一节要解决的就是这个更深层的问题。

### 14.1 Experience Pattern（体验模式）

**一个事件的核心，不是它的题材，而是玩家这一分钟到底在做什么决定。**

题材（门口的声音/窗外的影子/风吹动的东西）负责"这次穿的是什么皮肤"，Experience Pattern 负责"玩家这一刻在玩哪个小游戏"。两个事件题材完全不同，如果 Pattern 相同，玩家的决策体验仍然是重复的——这正是 Content QA v2 里"接近 vs 回避"占了 38% 的根本原因：题材各异，但 Pattern 高度集中。

已识别的 Pattern（不要求一次列全，以后可以继续补充）：

- 靠近 vs 保持距离
- 主动探索 vs 忽略
- 顺应本能 vs 克制
- 等待 vs 行动
- 冒险 vs 谨慎
- 接受 vs 拒绝
- 跟随 vs 独处
- 模仿 vs 坚持自己
- 分享 vs 独享
- 守护 vs 放弃
- 打断 vs 继续观察
- 主动回应 vs 被动接受（Owner Event Guideline 里已经出现过的具体案例）
- 警戒 vs 好奇（两个选项都主动，但情绪路径不同，不是"行动力度"的差异）

**使用原则**：同一个 Pattern 可以在不同题材里反复使用（这是正常的、必要的），但**整个内容池**里不应该让某一个 Pattern 占据压倒性比例。判断"这次要不要写这个 Pattern"，应该先看第14.5节的 Experience Pattern Audit，而不是只看"这个题材有没有写过"。

### 14.2 Event Workflow（事件设计流程）

旧的自然流程是"想一个故事 → 写两个选项 → 补数值"——这个顺序会导致两个问题：先固定题材，容易让 Pattern 无意识重复；数值放在最后补，容易让"保守选项"默认套用"calm+一个小值"这类公式，而不去思考这个数值到底该不该变化。

以后新增事件，按下面的顺序设计：

```
Experience Pattern（先决定玩家这一分钟在做什么决定，对照14.5节的 Pattern Audit 挑一个当前较少用的）
        │
        ▼
Animal Experience（在确定具体故事之前，先想清楚：这个 Pattern 落到猫/狗/兔身上，
                   分别是哪个感官在主导、哪种行为逻辑会不同——见第12.6节 Animal
                   Perspective 和 Owner Event Guideline 的"互换动物名字"自测法）
        │
        ▼
World Situation（现在才编具体的世界层事实——什么样的情境会自然产生
                 上面这个 Pattern + 这几种动物反应）
        │
        ▼
Perception（按物种写感知层文案，遵守第12节全部规范）
        │
        ▼
Choices（按选定的 Pattern 写选项，检查 Choice Balance：没有一眼看穿的最优解，
         也不是"两个选项其实是同一种态度"）
        │
        ▼
Feedback（现在时反馈文案，脱离上下文单独读、进入 Today's Memory 依然成立）
        │
        ▼
Stat（最后决定数值变化，此时做一次14.3节的 Value Economy Check）
```

**核心变化**：Experience Pattern 和 Animal Experience 被提到最前面——先决定"体验"，再决定"故事"；Stat 被留到最后，且不再是随手一补，而是一次显式的检查。

### 14.3 Value Economy（数值经济）

Content QA v2 发现：`calm`、`curiosity`、`bond`、`trust`、`dependence` 在选择层面几乎从未减少过——它们的行为更像 **Collection（收藏，只会累积）**，而不是 **Resource（资源，需要权衡取舍）**。

**这不是要求所有属性都必须双向变化**。一个属性长期只涨不跌，如果是经过思考后的结果，可以是合理的（比如"一天内建立的信任"本身确实更适合缓慢单向增长，而不是频繁大幅波动）。这一节要建立的是**思考过程本身**，不是数值上的强制平衡。

**Value Economy Check**（写 Stat 时必须问自己的问题）：
1. 这个属性在这个事件里，到底是不是一个需要玩家权衡的资源？还是我只是在"填一个看起来合理的正数"？
2. 保守/安全选项是不是又默认写成了"calm +一个小值"？有没有考虑过这里 calm 也可以不变、甚至下降？
3. 这个属性已经连续很多个事件都只朝一个方向变化了吗？这次是不是一个自然的机会，让它出现一次真实的反向变化？

这个 Check 在写完 Stat 后做一次即可，不需要为每个属性都强行找一个"变糟"的理由。

### 14.4 Character Range（性格弹性）

猫=克制观察、狗=兴奋外显、兔=应激退缩，这套区分度是 Batch 1 已经验证成功的设计，**应该继续保留作为每种动物的默认基调**，不是要推翻它。

但 Content QA v2 也发现：三种动物目前只展现了各自性格的"预期内"一面，缺少反差时刻——猫从未强烈外露过情绪，狗从未表现出专注冷静，兔从未在信任建立后放松/大胆过。长期只写"预期内"的反应，会让性格从"鲜明"滑向"刻板"。

**原则**：不是让动物 OOC（脱离性格设定），而是承认**同一种性格，在不同的具体情境下，会有不同的合理表现**——谨慎的猫遇到极度安全/极度熟悉的情境，也会短暂放下警惕；一直冷静观察的猫，遇到真正强烈的刺激，也应该有机会情绪失控一次。

**建议的落地方式（复用现有机制，不新增系统）**：这类"反差时刻"适合绑定在已有的 `precondition` 字段上（比如"社会层信任值达到较高阈值时，兔子才会出现一次罕见的放松/大胆事件"），或者放进 Rare/Secret 稀有度——用已有的字段实现"这是需要积累才会出现的特殊时刻"，而不是随时可能发生的常态，这样反差时刻本身也会显得更有分量。

### 14.5 Experience Pattern Audit（体验模式普查）

在既有的 Coverage Audit（统计题材/分类/稀有度/时段）之外，新增一套**长期并行统计**，每完成一个 Batch 都要做一次：

- **不统计**：故事/题材/分类。
- **统计**：每个事件的核心 Experience Pattern 是什么。
- **产出**：哪些 Pattern 已经过量，哪些几乎没有，供下一个 Batch 的 Backlog 优先选用。

这套统计和 Coverage Audit 是两个独立的维度——Coverage Audit 回答"内容够不够多样"，Experience Pattern Audit 回答"玩家的决策体验够不够多样"。两者都需要在每个 Batch 后运行。

### 14.6 Decision Psychology（决策心理）

**核心原则**：世界层的客观走向（客人是否离开、主人是否出门、天气是否持续、事件是否发生）不受玩家选择影响；玩家的选择只决定动物自身的状态、情绪、和与世界的关系如何变化。

**边界澄清**：`precondition` 门槛（如某个事件需要 Bond 达到一定数值才可能出现）不违反这条原则——门槛决定的是"能不能经历这段体验/以什么状态经历它"，是情境准入条件，不是"世界因为选择而改变了结果"。Character Range 的反差时刻如果绑定 precondition，同样不违反。

**玩家真正表达的是什么**：不是"我做出了正确/划算的决定"，是"此刻，我是一只怎样的猫/狗/兔子"。选项之间的差异不是收益高低，是"哪一种反应方式更接近这一刻的自己"。

**与其它系统的关系**：这条原则不是孤立的新规则，是 Event Layer（第7节，玩家只活在感知层，无从获取或操纵世界层真相）、Today's Memory（第10.5节，收尾句无差别肯定——正因为世界不变、只有主观经历不同，这句肯定才能对任何一种经历都成立）、Ending Framework（第13.4节，Death 被反复强调是"意外"而非"后果"）在"选择"这个维度上的同一套哲学。以后新增任何系统，只要涉及"玩家的选择会产生什么后果"，都应该先对照这条原则。

### 14.7 Surprise Audit（惊喜审查）

**目的**：不是"原创性评分"，是**内容开始同质化的预警信号**。Experience Pattern 明确允许（甚至要求）同一个 Pattern 被反复使用，所以"每个事件必须独一无二"不是这个 Audit 的目标；它要回答的是趋势性问题——"最近几个 Batch，新事件里找不到具体最近邻的比例是不是在下降"。

**方法（低主观性、可重复）**：不打分。对每一条新事件，只问一个"是/否 + 具名"的问题——"这个事件，最像事件库里已有的哪一个？" 能明确指出具体对照对象，就记录这对相似关系；想不出具体对照对象，就判定为"新鲜"，不需要给出理由或分数。这个方法把一个没有锚点的开放问题（新不新鲜），转化成一个有锚点的比较问题（像不像已知的某一个），评判者不需要发明标尺，稳定性和可重复性都更高。

**产出的使用方式**：产出是一份"事件A像事件B"的具名相似对列表，而不是分数——用来判断整体创作是否正在收敛，不用来否决某一个具体事件。

### 14.8 Emotional Transition Audit（情绪转场审查）

**为什么这是一个独立于 Emotion Curve 的问题**：事件池总体情绪比例回答的是一个**集合问题**（这个池子里 Comfort 出现了几次），答案不受抽取顺序影响；玩家实际体验到的是一个**序列问题**（连续两次抽到的事件，情绪标签相同的概率有多大），这个答案取决于触发时段/稀有度/类别的聚集效应，和总量是否均衡完全脱钩。一个事件池可以在总量上完全符合"情绪不过量"的目标，同时在连续抽样上依然让玩家连续遇到好几次同一种情绪——这不是理论假设，Owner 类 005/006、Rest 类 002/005 的连续同情绪都是集合总量正常、但序列层面已经失衡的真实案例。

**方法**：不手算概率公式（实际抽样受稀有度权重、动物权重、跨局降权共同影响，公式容易算错）。复用项目已经在用的 Node + vm 模拟脚本，跑数千局完整游玩，直接统计"连续两次触发的事件情绪标签相同"的实际发生频率。这个 Audit 不需要每个 Batch 都跑，可以按"每完成 2～3 个 Batch 跑一次"的节奏执行，因为它需要专门的模拟脚本，成本高于读表格统计。
